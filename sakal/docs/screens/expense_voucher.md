# Expense Voucher (Finance) — Design + Build

## Context

Use case: a service bill (electricity, water, internet, ...) needs to hit the books in the period it actually belongs to (January), even though it's entered and paid later (February) — textbook accrual accounting, creating a real payable against the supplier so a later Payment Voucher can settle it.

Two adjacent patterns don't quite fit: **Purchase Bill** does "recognize now, pay later" but only for goods received via a GRN (its whole GR/IR two-stage design exists to reconcile a GRN estimate against the real invoice — overhead a service bill doesn't need). **Journal Voucher** is free-form but its bill-linkage is opt-in/conditional — here, creating the bill is the entire point, always happening.

## Design — Odoo-style automatic tax, no manual Dr/Cr

Original spec asked for JV-style manual per-line Dr/Cr. Mid-discussion, after walking through Odoo's Vendor Bill mechanism in detail, the user agreed to drop manual Dr/Cr entirely in favor of Odoo's model: each line is **Account + Amount (always an implicit debit) + an optional Tax Group**. A normal (VAT-type) tax **adds** to the payable; a **withholding**-type tax **subtracts** from it — exactly like Odoo's tax repartition engine, but built on infrastructure this schema already had sitting dormant: `rim_tax_types.is_withholding` (seeded since migration 025, never consumed by any posting function until this one) and `rim_taxes.gl_expense_account_id` (a third GL slot alongside Output/Input VAT that never fit either pattern until now — clearly built for exactly this).

Key decisions:
- **Account picker excludes Cash/Bank/Customer/Supplier** — stricter than JV (which only excludes Cash/Bank), since Supplier is already the fixed computed line here.
- **Currency** anchors on the Supplier's own default (auto-fetched on pick, same pattern Purchase Order's own `_onSupplierSelected` already uses), freely re-selectable.
- **Balance is guaranteed by construction**, not a pre-check: Supplier's own CR line = (expense + VAT total) − (withholding total), forced to the literal accumulated `base_amount`, never re-derived via multiplication — same technique Purchase Bill's own PUR voucher (059) uses to keep `fn_post_voucher`'s DR=CR check from ever failing on floating-point rounding. Only real validation: **Net must be > 0**.
- **Bill-linkage is mandatory**, not opt-in like JV's — `inv_bill_no`/`inv_bill_date` on the Supplier line always come from the header's own Bill No/Bill Date.
- **Supplier is always `serial_no = 1`** in the posted voucher (mirroring Payment/Receipt Voucher's "line 1 is the fixed/anchor line" convention) even though it's computed last — achieved by prepending it to the line array before `fn_post_voucher` assigns serial numbers in array order.
- **`voucher_nature = 'EXPENSE'`** (a new value, user's explicit choice over reusing `PURCHASE`) and **Default Tax Group per account** (`rim_accounts.default_tax_group_id`, user's explicit choice to build now rather than defer) — both were presented as recommendations against the simpler path and the user chose the fuller-scope option for each.

## Build Session Notes

**Backend — `backend/migrations/107_expense_voucher.sql`**
- `rim_voucher_types_nature_check` extended from its actual CURRENT definition (confirmed via fresh grep across every migration, not the stale 017 baseline — the exact mistake already caught once during Contra Voucher) to add `EXPENSE`.
- **Two voucher_type_codes, not one**: `EXV` (the document's own numbering, used only by `fn_save_expense_voucher`) and `EXP` (the GL posting code, used only by `fn_approve_expense_voucher`'s `fn_post_voucher` call) — same split rationale as Purchase Bill's PINV/PUR and Material Issue's MREQ+MISS/MIC. **Caught before ever running**: an earlier draft used `EXV` for both, which would have made Approve silently consume/skip numbers from the document's own numbering sequence, since `ril_trans_no_seq` keys its counter on `(company, location, voucher_type_code)` alone.
- New tables `rih_expense_voucher_headers`/`rid_expense_voucher_lines` (real source-document tables, not a thin JV/Contra-style wrapper — see "why this needed real backend logic" below), `rim_accounts.default_tax_group_id`.
- `fn_save_expense_voucher` — DRAFT save/update, immutability-after-posted enforced.
- `fn_approve_expense_voucher` — period/backdate checks (5-arg form, reference-dated on `created_at`), expands each line's optional Tax Group into VAT-add / withholding-deduct GL lines (first real consumer of `is_withholding` anywhere in this schema), builds the Supplier CR line, composes `fn_post_voucher` (never writes to `rih_finance_headers`/`rid_finance_lines` directly).
- Menu seed (`FN-EXP`) — same two-part pattern (future clients via `fn_seed_client_modules.sql`, existing clients backfilled in this migration) as Contra's `FN-CTR`; same "doesn't grant access by itself" caveat.
- `backend/tests/107_expense_voucher_test.sql` — 14 pgTAP assertions: CHECK constraint + both voucher codes exist, a no-tax bill, a VAT-only bill (Supplier net = 116), a mixed VAT+WHT bill (Supplier net = 106, WHT posts CR to its own account), Net≤0 rejection, `v_pending_bills` visibility, and a regression test for the FK-ordering bug below. Paren-balanced (293=293).

**Why this needed real backend logic, unlike JV/Contra**: once line-level tax groups need server-side rate lookup and ADD/DEDUCT branching, the generic `fn_save_finance_voucher`/`fn_post_finance_voucher` pair can't be called directly from Flutter with a fully client-computed line array — tax computation has to be authoritative on the backend, same principle as GRN/Purchase Bill/Sales Invoice never trusting a client-sent tax amount. Structurally this module ended up closer to Purchase Bill (real header/line tables, a bespoke `fn_approve_*` composing the shared engine) than to JV/Contra.

**Real bugs caught in review, before ever running:**
1. **Voucher-code collision** (described above) — `EXV` used for both document numbering and GL posting. Fixed by adding a separate `EXP` posting code.
2. **FK violation on editing a DRAFT's own date** — `rid_expense_voucher_lines` has a real FK on `(client_id, company_id, trans_no, trans_date)`. The original `fn_save_expense_voucher` updated the header's `trans_date` to the new value, *then* deleted old lines filtering on that same new value — matching nothing (existing lines were still filed under the old date), and the header UPDATE itself would have failed outright with a foreign-key violation the moment a user changed the date on an existing draft with lines already saved. Fixed by capturing the old `trans_date` first and deleting under it *before* the header changes — the exact same fix GRN's own `fn_save_grn` already applies (`v_old_grn_date`), confirmed by reading GRN's code directly rather than assuming. Covered by pgTAP test 14.
3. Two `DropdownButtonFormField`s (header Currency, line Tax Group) have their value set both by their own `onChanged` *and* externally (Currency by `_onSupplierSelected`'s auto-fetch, Tax Group by `_onLineAccountSelected`'s auto-suggest from the account's own `default_tax_group_id`). A `FormField`'s `initialValue` is only read once at first build — the same "external state change doesn't visually resync" gotcha already caught twice this session for `SakalAutocomplete` (Contra Voucher's From/To swap). Fixed defensively with `key: ValueKey(...)` on both, forcing a clean remount whenever the value changes via either path — applied here proactively since the pattern was already known, not discovered by trial and error.

**Client-side tax preview, explicitly scoped**: the entry screen shows a live "Net Payable" preview using three new lightweight datasource methods (`getTaxGroupMemberTaxIds`/`getTaxRatesByIds`/`getTaxWithholdingFlags`) mirroring Purchase Order's own established `_taxGroupRatePct` pattern exactly, extended with the ADD/DEDUCT split. Explicitly labeled "(preview — confirmed at Approve)" in the UI — the backend remains the sole authoritative computation, avoiding any risk of client/server tax-logic drift.

**Offline**: new dedicated Drift cache tables (`ExpenseVoucherHeadersCache`/`ExpenseVoucherLinesCache`, schema v25) and `ExpenseVoucherLocalDs` — this module does NOT ride on the existing `FinanceVoucherHeadersCache` the way JV/Contra do, since it's a genuinely separate document type with its own header/line shape. Save-offline only; Approve stays online-only, same as every offline-capable module in this schema.

**Print**: new `EXPENSE_VOUCHER` document type registered in full (registry's 4 switches, `expense_voucher_default_template.dart`, `print_template_provider.dart`, `print_sample_data.dart`) — not a `VOUCHER`-type reuse like JV/Contra, since this document's header shape (Supplier/Bill No/Bill Date/Net Payable) doesn't fit the generic voucher fields.
